## Streaming style comparison

v3 added BufferList and StreamHub styles for incremental and real-time processing. The table below summarizes the relative cost of each style on an equivalent 502-quote dataset.

| Style          | Use case                        | Relative performance   | Latency per quote  |
| -------------- | ------------------------------- | ---------------------- | ------------------ |
| **Series**     | Batch / one-time calculation    | Baseline (fastest)     | N/A                |
| **BufferList** | Incremental / growing dataset   | ~10–20% overhead       | &lt;100 μs typical    |
| **StreamHub**  | Real-time / multi-indicator hub | ~20–30% overhead       | &lt;1 ms typical      |

### BufferList vs. Series

BufferList provides O(1) or O(log n) updates per quote for most indicators, avoiding a full recalculation on each addition.

```text
Series    (502 quotes): ~25 μs total
BufferList(502 quotes): ~30 μs total  (~20% overhead)
Per-quote latency:      ~60 ns average
```

**Indicators with O(1) per-quote updates:** SMA, EMA, RSI, Bollinger Bands — all use rolling-sum or Wilder-smoothing techniques that touch only the newest quote.

### StreamHub vs. Series

StreamHub adds observable state management on top of BufferList's incremental updates, enabling a single `quoteHub.Add()` call to propagate through multiple downstream hubs in one pass.

```text
Series    (502 quotes): ~25 μs total
StreamHub (502 quotes): ~32 μs total  (~28% overhead)
Per-quote latency:      ~64 ns average
Rollback (late Add):    ~2–5 μs for state rebuild
```

**Scaling characteristic:** Hub overhead is amortized across observers. Running 5 indicators on one hub costs roughly 1.5× the price of a single Series calculation, not 5×.

## Memory overhead

| Style          | Memory per instance               | Scaling factor         |
| -------------- | --------------------------------- | ---------------------- |
| **Series**     | ~4 KB (results only)              | Fixed for dataset size |
| **BufferList** | ~8 KB (buffers + results)         | Grows with lookback    |
| **StreamHub**  | ~12 KB (cache + state + results)  | Grows with lookback    |

## Latency targets for real-time systems

| Scenario                                | Target    | Typical performance  |
| --------------------------------------- | --------- | -------------------- |
| Single indicator per quote              | &lt;100 μs   | 60–80 μs             |
| 5 indicators on one hub per quote       | &lt;500 μs   | 300–400 μs           |
| Complex chain (EMA → RSI → Slope)       | &lt;200 μs   | 120–150 μs           |
| State rebuild after out-of-order quote  | &lt;5 ms     | 2–3 ms               |

    - Processing complete historical datasets

    - One-time or periodic batch calculations
    - Maximum throughput is the priority

    **Avoid when:**
    - Quotes arrive one at a time
    - Multiple indicators need synchronized updates
  </Card>
  <Card title="BufferList" icon="layers">
    **Choose when:**
    - Building up data incrementally
    - Running a single indicator on a growing dataset
    - Memory efficiency matters
    - No multi-indicator coordination needed

    **Avoid when:**
    - Multiple indicators must update together
    - You need rollback support for late-arriving data
  </Card>
  <Card title="StreamHub" icon="broadcast">
    **Choose when:**
    - Integrating a live WebSocket or market data feed
    - Multiple indicators must stay in sync
    - Low per-quote latency is critical
    - Rollback and state management are required

    **Avoid when:**
    - You only need a single indicator
    - All data is available upfront
  </Card>
</CardGroup>

## Optimization tips

### Limit result history with MaxListSize

For long-running BufferList or StreamHub processes you don't need to keep the full history, cap the list at a manageable size:

```csharp
// BufferList: keep only the last 1000 SMA values
SmaList smaList = new(20)
{
    MaxListSize = 1000
};

// StreamHub: cap the quote cache (all chained hubs inherit this)
QuoteHub quoteHub = new(maxCacheSize: 500);
SmaHub   smaHub   = quoteHub.ToSmaHub(20);
```

### Remove warmup periods before consuming results

Smoothing indicators (EMA, RSI, MACD, etc.) converge over time. Strip the unreliable warmup tail before downstream use:

```csharp
// auto-remove the library-recommended warmup count
IReadOnlyList<AdxResult> results = quotes
    .ToAdx(14)
    .RemoveWarmupPeriods();

// or specify a custom amount
IReadOnlyList<EmaResult> emaResults = quotes
    .ToEma(20)
    .RemoveWarmupPeriods(250);
```

<Warning>
  Providing only the _minimum_ required quote count is not a performance optimization. Smoothing indicators need 250+ preceding bars to reach full precision. Supply at least 750 bars (approximately three years of daily data) when precision matters.
</Warning>

### Chain instead of recalculating

If you need an indicator that depends on another, chain them rather than calling each independently. Chaining reuses the intermediate results in a single pass:

```csharp
// inefficient: OBV calculated twice
IReadOnlyList<ObvResult> obv = quotes.ToObv();
IReadOnlyList<RsiResult> rsi = quotes.ToObv().ToRsi(14);

// efficient: OBV calculated once, RSI reads from it
IReadOnlyList<RsiResult> rsi = quotes
    .ToObv()
    .ToRsi(14);
```

In a StreamHub, every downstream hub subscribes to the same propagation event — there is no duplication regardless of how many indicators you attach.

### Use Series for one-time analysis

If you only need indicator values once (backtest, report generation, CSV export), Series is the fastest option. Avoid spinning up a StreamHub for batch workloads.
